Royal Braids / Salon Shop - Deep Feature Inventory
==================================================

This file catalogs the current feature set for the Royal Braids salon platform.
It is intentionally broader than a marketing list: it explains feature purpose,
frontend/backend ownership, operational value, and implementation boundaries.

Active architecture summary
---------------------------
- Vercel serves the static public website and admin console from public/.
- Supabase owns Auth, Postgres data, migrations, constraints, RLS, and safe browser-facing reads.
- Render runs the trusted Node/Express backend API for privileged workflows, service-role writes, admin authorization, booking transactions, jobs, notifications, and Cloudinary signing.
- Cloudinary stores uploaded media when signed upload workflows are used.
- Resend and WhatsApp Cloud API are provider integrations for notifications when real sending is enabled.
- An external scheduler calls protected Render job endpoints with X-Job-Secret.
- Legacy Firebase production assets are archived under legacy/firebase-production-archive/ and are not active runtime dependencies.


1. Public website and brand experience
--------------------------------------
Purpose:
  Present Royal Braids as a professional, premium salon and guide visitors toward booking, gallery discovery, reviews, blog content, and contact.

Capabilities:
  - Responsive public homepage at public/index.html.
  - Branded splash screen and reveal behavior.
  - Premium Royal Braids visual identity with logo support.
  - Hero section with calls to action.
  - Smooth navigation for Home, Gallery, Services, Booking, Dashboard, Reviews, Blog, and Contact.
  - Mobile menu for narrow screens.
  - Dark/light mode support.
  - Theme preset support including brand color palettes.
  - SEO metadata and Open Graph metadata.
  - Favicon and logo image support.
  - Very-small-screen layout polish.
  - Public business contact and location display.

Primary files:
  - public/index.html
  - public/CSS/style.css
  - public/JS/script.js
  - public/JS/splash.js
  - public/JS/apply-client-config.js
  - public/client-config.js

Ownership notes:
  - Visual rendering belongs to public HTML/CSS/JS.
  - Business identity and public contact values should be configured in client-config.js.
  - Private business credentials must never be placed in public files.


2. White-label/client configuration
-----------------------------------
Purpose:
  Allow the salon brand, catalog, contact details, theme, and browser-safe integration values to be updated from one predictable file.

Capabilities:
  - Business name, slug, country, city, timezone, locale, and currency configuration.
  - Business logo text and logo image array.
  - Phone, email, WhatsApp, and contact notification display values.
  - Public Supabase URL/anon key configuration.
  - Public Render API base URL configuration.
  - Cloudinary public folder naming.
  - Theme preset and exact color override support.
  - Service category configuration.
  - Stylist configuration.
  - Service and sub-service catalog configuration.
  - Fallback content for public rendering where backend data is unavailable.

Primary file:
  - public/client-config.js

Safe to expose:
  - Public brand/contact information.
  - Public Supabase anon/publishable key.
  - Public Render API URL.
  - Public Cloudinary folder names.

Never expose:
  - Supabase service-role key.
  - Database passwords.
  - Resend API key.
  - WhatsApp access token.
  - Cloudinary API secret.
  - JOB_SECRET or scheduler secrets.

23. Safety and architecture constraints
---------------------------------------
Must remain true unless the architecture is intentionally changed:
  - public/ remains static and secret-free.
  - Browser code uses only public Supabase anon config and public Render API URL.
  - Supabase service-role key is stored only on Render.
  - RLS protects browser-facing data access.
  - Render verifies Supabase access tokens.
  - Render owns privileged booking, waitlist, admin, notification, upload, and security workflows.
  - Admin permissions are centralized through admin_users.
  - Provider secrets are stored only in server-side environment variables.
  - Firebase archive remains historical reference only.
  - Operational records should prefer status/audit changes over destructive deletion.
